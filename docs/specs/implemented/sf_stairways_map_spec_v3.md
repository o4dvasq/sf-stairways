# SF Stairways Map Tracker — Claude Code Spec v3
## Final spec — includes in-app walk logging via GitHub API

---

## Overview

A static, browser-based map of all ~369 public stairways in San Francisco with a
curated personal target list and walk progress tracking. Two ways to log a walk:
1. **In-app:** click a marker → "Mark as Walked" → saves directly to GitHub
2. **Claude Code:** tell Claude which stairs you walked, it commits the update

No backend. No build step. Runs entirely on GitHub Pages.

## Phase 0: Data Acquisition

### `scripts/scrape_stairways.py`

Produces `data/all_stairways.json`. Run once; commit the result.

**Strategy per stairway — try in order:**

1. Scrape detail page at `https://www.sfstairways.com/stairways/{slug}`
   - Look for lat/lng in: Google Maps embed params, JS variables (`lat =`, `center: {lat:`), JSON-LD
2. Fallback: Nominatim geocode via `https://nominatim.openstreetmap.org/search?q={name}+San+Francisco+CA&format=json&limit=1`
   - Tag record with `"geocode_source": "nominatim"`
3. Last resort: `lat: null, lng: null` — no marker rendered, no error

**Crawl rules:**
- 0.5s sleep between sfstairways.com requests
- 1.0s sleep between Nominatim requests
- Skip closed/demolished stairways (tag `"closed": true`, exclude from map)
- Print progress: `[42/369] Vulcan Stairway — coords from page`

**Output schema:**
```json
{
  "id": "vulcan-stairway",
  "name": "Vulcan Stairway",
  "neighborhood": "Corona Heights",
  "lat": 37.7635,
  "lng": -122.4420,
  "height_ft": 115,
  "closed": false,
  "geocode_source": "page",
  "source_url": "https://www.sfstairways.com/stairways/vulcan-stairway"
}
```

## In-App Edit Flow

### "Mark as Walked" (target marker popup)
Clicking the button expands the popup in-place with an inline mini-form:

```
✅ Mark as Walked

Date: [ March 8, 2026  ▾ ]     ← date input, defaults to today
Steps: [ 148            ]       ← pre-filled if known, editable

[ Save ]  [ Cancel ]
```

On Save:
1. Update `target_list.json` in memory
2. Write to GitHub (see GitHub API section below)
3. Re-render the marker as green, update stats bar
4. Close popup

### "Edit" (walked marker popup)
Same inline form, pre-filled with current values. Allows correcting date or step count.

### "+ Add to Target List" (gray marker popup)
Inline mini-form:

```
+ Add to Target List

Steps (if known): [ ___ ]
Notes:            [ ___ ]

[ Add ]  [ Cancel ]
```

On Add:
1. Copy record from `all_stairways.json` into `target_list.json` in memory
2. Add `step_count`, `notes`, `walked: false`, `date_walked: null`
3. Write updated `target_list.json` to GitHub
4. Re-render marker as red target dot

## GitHub API Integration

### How it works
The app uses the GitHub Contents API to read and write `data/target_list.json`
directly from the browser. This requires a GitHub Personal Access Token (PAT)
with `repo` scope (or `contents: write` for fine-grained tokens).

No server involved. The token is stored in `localStorage` under key `gh_token`.
Repo owner and repo name are hardcoded in a config block at the top of `index.html`.

### Config block (top of JS section in index.html)
```javascript
const CONFIG = {
  githubOwner: 'OWNER',      // ← Claude Code: leave as placeholder
  githubRepo:  'sf-stairways',
  dataPath:    'data/target_list.json',
  branch:      'main'
};
```

### Write flow
```javascript
async function saveTargetList(updatedList) {
  const token = localStorage.getItem('gh_token');
  if (!token) { showTokenModal(); return; }

  // 1. Get current file SHA (required by GitHub API for updates)
  const getRes = await fetch(
    `https://api.github.com/repos/${CONFIG.githubOwner}/${CONFIG.githubRepo}/contents/${CONFIG.dataPath}`,
    { headers: { Authorization: `token ${token}` } }
  );
  const { sha } = await getRes.json();

  // 2. PUT updated content
  const content = btoa(JSON.stringify(updatedList, null, 2));
  await fetch(
    `https://api.github.com/repos/${CONFIG.githubOwner}/${CONFIG.githubRepo}/contents/${CONFIG.dataPath}`,
    {
      method: 'PUT',
      headers: {
        Authorization: `token ${token}`,
        'Content-Type': 'application/json'
      },
      body: JSON.stringify({
        message: `Log walk: ${new Date().toISOString().split('T')[0]}`,
        content,
        sha,
        branch: CONFIG.branch
      })
    }
  );
}
```

### Error handling
- 401 Unauthorized → clear stored token, re-show token modal with message "Token rejected — please re-enter"
- 409 Conflict (SHA mismatch) → show: "Someone else updated the file — please reload and try again"
- Network error → show: "Couldn't reach GitHub — change saved locally, retry when online"
  - Keep a `pendingChanges` flag in memory; offer a "Retry" button in the header

```javascript
const [targetList, allStairways] = await Promise.all([
  fetch('./data/target_list.json').then(r => r.json()),
  fetch('./data/all_stairways.json').then(r => r.json())
```

- `target_list.json` failure → error banner, map non-functional
- `all_stairways.json` failure → "Show All SF" button disabled with tooltip, target list works normally
- Records with `lat: null` or `lng: null` → silently skipped, no marker

## README

1. **Live URL:** `https://[username].github.io/sf-stairways/`
2. **First-time setup:**
   - Fork/clone repo, enable GitHub Pages (Settings → Pages → main branch / root)
   - Open the live URL, click ⚙, paste your GitHub PAT
   - Token is stored in your browser — set it once per device
3. **Log a walk in the field:** Tap a red marker → "Mark as Walked" → confirm date → Save
4. **Bulk update via Claude Code:** Tell Claude which stairs you walked and the date;
   Claude edits `target_list.json` directly and commits
5. **Add a stairway to your target list:** Tap any gray dot → "+ Add to Target List"
6. **Local dev:** `python3 -m http.server 8080` → `localhost:8080`
   (GitHub API writes work locally too — same token)
7. **Regenerate city data:** `pip install requests beautifulsoup4 && python scripts/scrape_stairways.py`
